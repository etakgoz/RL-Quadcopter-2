Controlling the Quadcopter¶

We provide a sample agent in the code cell below to show you how to use the sim to control the quadcopter. This agent is even simpler than the sample agent that you'll examine (in agents/policy_search.py) later in this notebook!

The agent controls the quadcopter by setting the revolutions per second on each of its four rotors. The provided agent in the Basic_Agent class below always selects a random action for each of the four rotors. These four speeds are returned by the act method as a list of four floating-point numbers.

For this project, the agent that you will implement in agents/agent.py will have a far more intelligent method for selecting actions!

import random

class Basic_Agent():
    def __init__(self, task):
        self.task = task
    
    def act(self):
        new_thrust = random.gauss(450., 25.)
        return [new_thrust + random.gauss(0., 1.) for x in range(4)]

The Task¶

A sample task has been provided for you in task.py. Open this file in a new window now.

The __init__() method is used to initialize several variables that are needed to specify the task.

  • The simulator is initialized as an instance of the PhysicsSim class (from physics_sim.py).
  • Inspired by the methodology in the original DDPG paper, we make use of action repeats. For each timestep of the agent, we step the simulation action_repeats timesteps. If you are not familiar with action repeats, please read the Results section in the DDPG paper.
  • We set the number of elements in the state vector. For the sample task, we only work with the 6-dimensional pose information. To set the size of the state (state_size), we must take action repeats into account.
  • The environment will always have a 4-dimensional action space, with one entry for each rotor (action_size=4). You can set the minimum (action_low) and maximum (action_high) values of each entry here.
  • The sample task in this provided file is for the agent to reach a target position. We specify that target position as a variable.

The reset() method resets the simulator. The agent should call this method every time the episode ends. You can see an example of this in the code cell below.

The step() method is perhaps the most important. It accepts the agent's choice of action rotor_speeds, which is used to prepare the next state to pass on to the agent. Then, the reward is computed from get_reward(). The episode is considered done if the time limit has been exceeded, or the quadcopter has travelled outside of the bounds of the simulation.

In the next section, you will learn how to test the performance of an agent on this task.

The Agent¶

The sample agent given in agents/policy_search.py uses a very simplistic linear policy to directly compute the action vector as a dot product of the state vector and a matrix of weights. Then, it randomly perturbs the parameters by adding some Gaussian noise, to produce a different policy. Based on the average reward obtained in each episode (score), it keeps track of the best set of parameters found so far, how the score is changing, and accordingly tweaks a scaling factor to widen or tighten the noise.

import sys
import csv
import numpy as np
from agents.agent import DDPG
from task import TakeOffTask

# Modify the values below to give the quadcopter a different starting position.
runtime = 5.                                     # time limit of the episode
file_output = 'takeoff_data.txt'                 # file name for saved results

# Setup
target_pos = np.array([0., 0., 10.])
init_pose = np.array([0., 0., 1., 0., 0., 0.])
task = TakeOffTask(runtime=runtime, init_pose=init_pose, target_elevation=10.0)

agent = DDPG(task)

done = False
labels = ['time', 'x', 'y', 'z', 'phi', 'theta', 'psi', 'x_velocity',
          'y_velocity', 'z_velocity', 'phi_velocity', 'theta_velocity',
          'psi_velocity', 'rotor_speed1', 'rotor_speed2', 'rotor_speed3', 'rotor_speed4','reward']

n_episodes = 100

# Variables to record episode performance
results = {x : [] for x in labels}
best_episode = -1
best_reward = 0
rewards = np.zeros((n_episodes))
moving_reward_average = 0 # for the last 10 episodes


# Run the simulation, and save the results.
with open(file_output, 'w') as csvfile:
    writer = csv.writer(csvfile)
    writer.writerow(labels)
    for i in range(0, n_episodes):
        state = agent.reset_episode()
        while True:
            # predict
            rotor_speed = agent.act(state)
            rotor_speeds = [rotor_speed[0], rotor_speed[0], rotor_speed[0], rotor_speed[0]]
            
            # apply rotor speeds
            next_state, reward, done = task.step(rotor_speeds)
                        
            # training step
            agent.step(rotor_speed, reward, next_state, done)
            
            # add current rewards to the step
            rewards[i] += reward
            
            # logging
            to_write = [task.sim.time] + list(task.sim.pose) + list(task.sim.v) + list(task.sim.angular_v) + list(rotor_speeds) + list([reward])
            for ii in range(len(labels)):
                results[labels[ii]].append(to_write[ii])
            writer.writerow(to_write)

            if done:
                writer.writerow("--Episode End--")
                
                # update best episode and save episode number/reward
                if best_episode < 0:
                    best_reward = rewards[i]
                elif best_reward < rewards[i]:
                    best_reward = rewards[i]
                
                moving_reward_average = np.mean(rewards[i-10:i]) if i > 10 else 0
                    
                print("\rEpisode = {:4d}, reward = {:7.3f}, best_reward = {:7.3f}, moving_reward_average (last 10 eps) = {:7.3f}".format(i, rewards[i], best_reward, moving_reward_average), end="")

                break
            else:
                state = next_state
        sys.stdout.flush()
        
        

Episode =   99, reward = -10.917, best_reward = -10.917, moving_reward_average (last 10 eps) = -10.91359

Reflections¶

Question 1: Describe the task that you specified in task.py. How did you design the reward function?

Answer: I defined the TakeOff task. The class constructor gets inital position and target elevation to take off. I defined action_size of the task to be 1 to predict one rotor speed for each 4 rotors. Also, I slightly modified the done condition so the task also ends when the quadcopter rises above the target elevation.

I discussed the reason for my choices in question 4 as well, but without these modifications I found the training experience to be frustrating. I was not able to get stable improvements. I've experimented with a combination of different factors for the reward function: euclidean distance, angular orientation, rotor speed variance, speed in different axis, clamping rewards.

My final reward function turned out be below:

# distance from target elevation
        reward = -0.1 * abs(self.target_pos[2] - self.sim.pose[2])

        # absolute reward when target elevation is reached
        reward = 10 if math.floor(self.sim.pose[2]) > math.floor(self.target_pos[2]) else reward

        return reward

It instructs the quadcopter stay close to target elevation as close as possible and rewards it positively when the target elevation is reached.

  • What was your final choice of hyperparameters (such as $\alpha$, $\gamma$, $\epsilon$, etc.)?

Answer:

I have implement my agent using Deep Deterministic Policy Gradient algorithm since both the input and output states were continous. I've used the template code provided by Udacity.

To make sure my agent is working I decided to solve a little bit simpler yet similar problem at Open AI Gym: Pendulum. I was able to solve it using my agent after adjusting the hyperparameters. While solving the problem, I noticed for some reason regularization did not work well on the critic model, so I've only put some dropout on actor model. (Perhaps this was related to my implementation, I've lefted regularization on critic model - normalization and droput commented out.)

My choice of hyperparameters:¶

Gamma = 0.98, Tau = 0.002,

Noise Process:¶

mu = 0, theta = 0.25, sigma = 0.2

Replay Memory:¶

Buffer Size: 10000, Batch Size: 256

Actor network architecture:¶

drouput rate = 0.2 optimizer Adam, Learning Rate = 0.0009

Layer 1: 32 neurons, RELU activation, droput Layer 2: 64 neurons, RELU activation, droput Layer 3: 32 neurons, RELU activation, droput Output Layer: Action Size neurons, Sigmoid activation

Critic network architecture:¶

Optimizer Adam, Learning Rate = 0.001 (default rate) Action Pathway: Layer 1: 32 neurons, RELU activation Layer 2: 64 neurons, RELU activation

State Pathway: Layer 1: 32 neurons, RELU activation Layer 2: 64 neurons, RELU activation

Combined Layer + RELU activation Additional Layer: 32 neurons, RELU activation Output Layer: 1 neuron to produce q values

Question 4: Briefly summarize your experience working on this project. You can use the following prompts for ideas.

  • What was the hardest part of the project? (e.g. getting started, plotting, specifying the task, etc.)
  • Did you find anything interesting in how the quadcopter or your agent behaved?

Answer:

In the beginning, the hardest part was getting started and making sure the agent learns. After clearing that, the hardest part was trying to define a suitable task and reward function.

After some initial unseccusful experiments with the quadcopter project, I tried to solve the Open AI Gym Pendulum problem first to make sure my agent learns. I was able to find some other solutions for that problem using DDPG as well and compared my network/hyperparameters with theirs. Some projects that I have found helpful:

  1. TORCS project: https://yanpanlau.github.io/2016/10/11/Torcs-Keras.html
  2. Solution to Pendulum: https://towardsdatascience.com/reinforcement-learning-w-keras-openai-actor-critic-models-f084612cfd69
  3. Another Solution to Pendulum: https://github.com/lirnli/OpenAI-gym-solutions/tree/master/Continuous_Deep_Deterministic_Policy_Gradient_Net

After some hyperparameter adjustments I was able to solve Pendulum within reasonable number of epochs. I added my solution to the end of the notebook too.

Then I focused on solving the TakeOff Task. Intially, I copied the original task to define the TakeOff task and did not modify the done condition. Later, I realized it is quite hard to design a reward function which will enforce agent to learn stabilizing rotor speeds and reach the target take off position (and stay there). I made two changes in my task definition to simplify the problem:

  1. Changed the done condition so the episode ends when the quadcopter reaches above takeoff height: This simplified the problem since without this change the TakeOff task essentially becomes taking off, and hovering until the episode timeouts.

  2. I decreased the action_size to 1 after noticing the rotor_speeds should be equal to each other for vertical take off. (I know this might not be the case if there is wind, etc.). By doing this, I stopped worrying about teaching quadcopter to stabilize its rotor speeds so it can move vertically. I know this is a simplication of the problem by incorporating my own physics knowledge but otherwise I found it very hard to teach quadcopter to takeoff and keep its elevation within reasonable number of epochs. (Though I think it is possible to do it with a clever reward function and enough training time.)
